2//! # Introduction
3//!
4//! WTFM is <a href="https://en.wikipedia.org/wiki/RTFM">RTFM</a> in Rust.
5//!
6//! To master an ecosystem as broad and deep as Rust in a short period of time,
7//! we might want to turn R into W, dive deep by writing our own
8//! [doctests](https://doc.rust-lang.org/rustdoc/write-documentation/documentation-tests.html) iteratively until we find the solution to our problem.
9//!
10//! ```
11//! assert_eq!("WTFM", "RTFM".replace("R", "W"));
12//! ```
13//! Doctests can be good "paper trails" of our thought process.
14//!
15//! We can also learn about the throught process of other crates by "reviewing"
16//! them with our own doctests.
17//!
18//! # Professional rustdoc rabbit holes
19//! Be prepared to fall into rabbit holes of rustdoc and get out
20//! where and when you need to. This is a skill to be learned.
21//! We are programmers, not librarians.
22//!
23//! <https://rust-lang.github.io/api-guidelines/documentation.html>
24//!
25//! # Doctest
26//! If `cargo test --doc` passed, we will have have output like
27//! ```text
28//! all doctests ran in 0.70s; merged doctests compilation took 0.33s
29//! ```
30//! There is no need to check each one of them in playground but we can explore
31//! further by playwith them in the plalyground (that is why we call it playground.)
32//!
33//! Since doctests are merged and compiled into one binary, we want to make sure
34//! they don't interfere with each other. Since doctests are not the real part
35//! of the code, we can have cleanup code in the libray body or unit test body.
36//! We can see how the source code progress this way.
37//!
38//! # [assert!] driven development
39//! Be prepared to [RTFM](https://doc.rust-lang.org/rustdoc/write-documentation/documentation-tests.html) [std].
40//!
41//! Let's start with an extremely trivial rust program using
42//! macro [assert!] and primitive [true].
43//! ```
44//! assert!(true);
45//! ```
46//!
47//! We can wrap it with a function `assert_true_v1` and call it in the
48//! same doctest block:
49//! ```
50//! fn assert_true_v1() {
51//!     assert!(true);
52//! }
53//! assert_true_v1();
54//! ```
55//! To make it a little be more complicated we can wrap [true] with
56//! a function as well.
57//! ```
58//! fn return_true() -> bool { true }
59//! assert!(return_true());
60//! ```
61//! The version 2 of `asser_true` can be
62//!
63//! ```compile_fail
64//! fn assert_true_v2() {
65//!     assert!(return_true());
66//! }
67//! ```
68//! The reason the doctest above failed is `return_true` is out of scope.
69//!
70//! Let's put function [return_true] in the crate,
71//! following works in `cargo test --doc` but
72//! would fail in playground because playground can't resolve our crate.
73//! ```
74//! use wtfm_rs::return_true;
75//! fn assert_true_v2() {
76//!     assert!(return_true());
77//! }
78//! ```
79//! We can add similar test to the unit test of this crate (see the source code)
80//! `cargo test` will capture all of them.
81//! ```text
82//!  cargo test
83//!    Finished `test` profile [unoptimized + debuginfo] target(s) in 0.00s
84//!     Running unittests src/lib.rs (target/debug/deps/wtfm_rs-ee0d1b96bcb289bd)
85//!
86//! running 1 test
87//! test assert_true_v3 ... ok
88//!
89//! test result: ok. 1 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s
90//!
91//!   Doc-tests wtfm_rs
92//!
93//! running 4 tests
94//! test src/lib.rs - (line 51) ... ok
95//! test src/lib.rs - (line 73) ... ok
96//! test src/lib.rs - (line 58) ... ok
97//! test src/lib.rs - (line 11) ... ok
98//!
99//! test result: ok. 4 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out; finished in 0.00s
100//!
101//!
102//! running 1 test

110//! # [println!] driven development
111//! If we add [println!] to the doctest
112//! ```
113//! use wtfm_rs::return_true;
114//! fn assert_true_v3() {
115//!     assert!(return_true());
116//!     println!("{}", return_true());
117//! }
118//! ```
119//! `cargo test` will not have any println feedback.
120//! ```text
121//! test src/lib.rs - (line 110) ... ok
122//! ```
123//!
124//! So `println!` driven development wouldn't work with doctests.
125//!
126//! To do `println!` driven development, we can use example crates.
127//! ```text
128//! cargo doc --examples
129//! ```
130//! ```text
131//! cargo run --example example-assert-true
132//!    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.00s
133//!     Running `target/debug/examples/example-assert-true`
134//! true
135//! ```

137//! # External crates
138//! Rustdoc is organized by crate and we can also bring external crates in
139//! via `Cargo.toml`
140//!
141//! ```toml
142//! [package]
143//! name = "wtfm-rs"
144//! version = "0.1.0"
145//! edition = "2024"
146//!
147//! [dependencies]
148//! wtfm-rs-hello-world = { git = "https://github.com/wtfm-rs/wtfm-rs.github.io", branch = "hello-world", version = "0.1.0" }
149//! ```
150//!
151//! In this case, we pull [wtfm_rs_hello_world] crate from
152//! <https://github.com/wtfm-rs/wtfm-rs.github.io/tree/hello-world>.
153//!
154//! We can use the function `hello_world()` from it
155//! ```text
156//! pub use wtfm_rs_hello_world::hello_world;
157//! ```
158//! and use it as `wtfm_rs::hello_world`.
159//!
160//! ```
161//! use wtfm_rs::hello_world;
162//! assert_eq!(hello_world(), "Hello, world!");
163//! ```
164//! This will fail in playwright as before, but will pass `cargo test`.
165//!
166//! # Get the ouput from echo Hello, world!
167//! ```
168//! use std::process::Command;
169//! let output = Command::new("echo")
170//!        .arg("Hello,")
171//!        .arg("world!")
172//!        .output()
173//!        .expect("Failed to execute command");
174//! let output_1 = String::from_utf8_lossy(&output.stdout).to_string();
175//! assert_eq!(format!("{}", output_1), "Hello, world!\n");
176//! let output_2 = String::from_utf8(output.stdout).expect("Format error");
177//! assert_eq!(format!("{}", output_2), "Hello, world!\n");
178//!
179//! ```
180//! [std::process::Command]
181//!
182//! [String::from_utf8]
183//!
184//! [String::from_utf8_lossy]
185

187    true
